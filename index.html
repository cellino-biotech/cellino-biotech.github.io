Cellino software development

A guide to configuring your local environment.

Intro

Prerequisites

The main prerequisites for our development environment are git and pyenv. These tools help manage code repositories and local Python environments, respectively. Each developer also needs to configure gcloud as a way to authenticate access to our cloud services. Package managers, such as homebrew or chocolatey, are highly recommended to streamline installation procedures.

Windows

Chocolatey

If not already installed, it is highly recommended to use chocolatey for general package management in Windows. Open an administrative shell (PowerShell) and download and run the installation script:

Set-ExecutionPolicy Bypass -Scope Process -Force; [System.Net.ServicePointManager]::SecurityProtocol = [System.Net.ServicePointManager]::SecurityProtocol -bor 3072; iex ((New-Object System.Net.WebClient).DownloadString('https://community.chocolatey.org/install.ps1'))

Git

Download the git installer and follow the guided installation. Most users prefer the default settings, though you may consider customizing the default text editor and shell application. Many current Cellino developers opt to use Git Bash, especially if the individual is more familiar with bash and Unix tools.

Pyenv

Though lacking some of the useful features found in pyenv, pyenv-win is still a great tool for managing different Python installations on your local machine. Install using chocolatey:

choco install pyenv-win

Now you see why we pushed chocolatey

To install Python distributions, with version 3.9.6 as an example:

pyenv install 3.9.6

Check to see all Python versions on local machine and configure scope settings:

print list of installed Python versions

pyenv versions

set Python 3.7 as the global executable

pyenv global 3.7.8

set Python 3.9 as the local executable for the current working directory

pyenv local 3.9.6

Unfortunately, pyenv-win does not support virtual environment management. To create a virtual environment in Windows, first navigate to the venv directory of the project repo and, after switching to the desired Python version, run:

python -m venv <venv-name>

Each time you start a shell, the virtual environment will have to be activated:

projectdir\venv\venv-name\Scripts\activate
